# Capacitor Architecture — one codebase, two surfaces

**Status:** authoritative reference. Read this before adding a new "web vs
native" branch anywhere in the code.

---

## The rule

MinDrop ships from **one React codebase**. The web at `mindrop.in` and the
Android APK share:

- every route under `src/routes/`
- every component under `src/components/`
- every hook under `src/hooks/`
- every server function under `src/lib/**/*.functions.ts`
- the same Supabase client, TanStack Query cache, and business logic

**Only the entry point differs**, and even that is handled by the framework —
we do not maintain two `main.tsx` files, two `App.tsx` files, or two Vite
configs.

---

## Why two surfaces need slightly different entries

TanStack Start's normal build is a Cloudflare Worker that server-renders each
route on demand. That works great for a web crawler hitting
`https://mindrop.in/pricing` — it gets fully-rendered HTML with real meta
tags.

Capacitor's Android WebView can't run a Worker. It needs a plain,
route-agnostic `index.html` on disk that boots the SPA and lets the client
router take over. That's it — the WebView doesn't need SSR because
`mindrop.in` isn't in the store listing, the APK is.

## How we solve it: Android-only `spa` mode

In `vite.config.ts`:

```ts
tanstackStart: {
  server: { entry: "server" },
}
```

At web publish build time, TanStack Start runs the normal Nitro build →
Cloudflare Worker output + client assets in `dist/client/`.
**This is what mindrop.in serves.**

For Android builds, `npm run build:android` runs the same framework build,
copies the generated client bundle, and writes a static **route-agnostic SPA
shell** directly. It does not start TanStack's preview server and does not
import any server build entry. The result is written to
**`.output/public/index.html`**.

That single extra file is what Capacitor packages into the APK.

```text
.output/public/
├── index.html          ← SPA shell (used by Capacitor)
├── assets/…            ← hashed JS/CSS chunks (used by both)
├── firebase-messaging-sw.js
└── robots.txt
```

The generated client source remains:

```text
dist/client/
├── assets/…            ← generated by TanStack Start + Nitro
└── static files         ← copied into .output/public for Android
```

`capacitor.config.ts` points at `webDir: ".output/public"`. `npx cap sync
android` copies the whole directory into
`android/app/src/main/assets/public/`.

## Request flows

### Web — SSR
```text
browser → mindrop.in → Cloudflare Worker
         └─ src/server.ts wrapper
            └─ TanStack Start handler
               └─ __root.tsx + matched route
                  └─ loader() runs on the server
                  └─ returns full HTML + hydration payload
```

### Native — SPA
```text
Android WebView → file://…/assets/public/index.html   (SPA shell)
                → boots @tanstack/react-start client
                → router mounts (memory-history-like from initial URL "/")
                → navigates to /splash (see isNativeApp() redirect on `/`)
                → loaders run *in the WebView*, calling server functions
```

### Server function call — web
```text
component → useServerFn(fn) → fetch("/_serverFn/<hash>", …)
                            → same-origin Worker handles it
                            → runs .handler() with real env vars
```

### Server function call — native
```text
component → useServerFn(fn) → fetch("/_serverFn/<hash>", …)
                            → window.fetch wrapper (installNativeApiForwarder)
                              rewrites path → https://mindrop.in/_serverFn/<hash>
                            → Cloudflare Worker handles it
                            → auth middleware reads Supabase bearer token
                              (client middleware in src/start.ts still attaches it)
```

The fetch wrapper is installed once in `__root.tsx`:

```ts
useEffect(() => { installNativeApiForwarder(); … }, []);
```

It's a no-op on web. On native it rewrites `/_serverFn/*` and `/api/*` to
`VITE_API_ORIGIN` (default `https://mindrop.in`). Absolute URLs (Supabase,
Google, Firebase) pass through untouched.

### Authentication
Auth is Supabase-based on both surfaces. The Supabase JS client stores its
session in whatever the WebView's `localStorage` gives it, and every server
function call carries the bearer token via the existing client middleware in
`src/start.ts`. Nothing native-specific needed for the auth flow itself.

### Native plugin flow (custom Kotlin bridges)
```text
TS: import { PlacesBridge } from "./native/placesBridge"
    ↓ registerPlugin("PlacesBridge") from @capacitor/core
Capacitor WebView bridge (JSON messages)
    ↓
PlacesBridgePlugin.kt (native/android/places/…)
    ↓
Android SDK (GeofencingClient, AlarmManager, etc.)
```

See `docs/CAPACITOR_BUILD.md` for the `MainActivity.java` `registerPlugin(…)`
snippet — Capacitor 8 does NOT auto-register plugins that live outside npm
packages.

## Platform layer

App code MUST NOT touch `@capacitor/*`, `window.Capacitor`, or
`navigator.userAgent` directly. Import from `@/lib/platform`:

```ts
import { isNative, isWeb, isAndroid, storage, serverApi } from "@/lib/platform";
```

Contract:

| helper                         | web                       | native                                       |
| ------------------------------ | ------------------------- | -------------------------------------------- |
| `isNative()` / `isNativeApp()` | `false`                   | `true`                                       |
| `isWeb()`                      | `true`                    | `false`                                      |
| `isAndroid()` / `isIOS()`      | `false`                   | true on that platform                        |
| `isCapacitor()`                | `false`                   | `true`                                       |
| `serverApi()`                  | `""`                      | `VITE_API_ORIGIN` or `https://mindrop.in`    |
| `installNativeApiForwarder()`  | no-op                     | installs `window.fetch` wrapper              |
| `storage.get/set/remove`       | `localStorage`            | `@capacitor/preferences`, falls back to LS   |

New helpers should extend this module rather than sprouting `if
(window.Capacitor)` branches in feature code.

## Native feature compatibility

The architecture supports all planned MinDrop features without another
rebuild. Each is implemented by installing the relevant Capacitor plugin (or
custom Kotlin bridge) and wrapping it in `src/lib/platform/`:

| feature                        | plumbing                                             |
| ------------------------------ | ---------------------------------------------------- |
| Exact alarms                   | Custom Kotlin (`AlarmManager.setAlarmClock`)         |
| Local notifications            | `@capacitor/local-notifications`                     |
| Push (FCM)                     | `@capacitor-firebase/messaging` (already installed)  |
| Notification Listener          | Custom Kotlin `NotificationListenerService`          |
| Background location, geofences | Custom `PlacesBridgePlugin.kt` + `GeofenceReceiver`  |
| Wake locks                     | Kotlin `PowerManager.WakeLock` inside foreground svc |
| Boot completed receiver        | Kotlin `BroadcastReceiver` in AndroidManifest        |
| Voice recording                | `@capacitor/voice-recorder` plugin                   |
| Google Drive backup            | Existing REST flow (works in WebView unchanged)      |
| Offline reminder execution     | Room DB mirror + native AlarmManager                 |
| Cashfree                       | Web SDK works; native SDK bridged when needed        |
| Play Billing                   | Kotlin bridge (`BillingClient`)                      |
| Subscription plans             | Same DB / server functions as web                    |
| Deep links, app links          | `capacitor.config.ts` + `intent-filter` in manifest  |
| Widgets, Wear OS               | Standalone Kotlin modules under `android/`           |

None of these require re-architecting the JS side.

## Build commands

| command                     | what it does                                                                |
| --------------------------- | --------------------------------------------------------------------------- |
| `npm run dev`               | Vite dev server (SSR). Works for both web and Android live-reload.          |
| `npm run build`             | Web/publish production build. No Capacitor prerender shell.                 |
| `npm run build:android`     | Android production build. Emits `.output/public/index.html` SPA shell.   |
| `./rebuild.sh`              | `git pull` → `npm install` → `npm run build:android` → `cap sync` → Kotlin mirror. |
| `npx cap sync android`      | Copies `.output/public/` into `android/app/src/main/assets/public/`.     |
| `npx cap open android`      | Opens Android Studio.                                                       |
| `./android-doctor.sh`       | Verifies local JDK / SDK / gradle prerequisites.                            |

Web publishing and Android packaging are intentionally separate build targets
so Lovable publish never waits on the Capacitor prerender flow.

---

## What must NOT drift

- **No `src/entry.capacitor.tsx`**, no `capacitor/index.html`, no second Vite
  config. The framework does this for us.
- **No duplicate `App.tsx` / route trees.** If you're tempted to add
  `if (isNative)` branches at the route level, add a `useEffect` redirect
  inside the shared route instead (see `src/routes/index.tsx`).
- **No direct `window.Capacitor` reads outside `src/lib/platform/`.**
- **No baking secrets into `VITE_*` vars.** Server functions read env inside
  their `.handler()` — that stays true whether the caller is web or native.

## What must stay in sync

If you rename `src/routes/pricing.tsx`, the Android app picks up the change
on the next `./rebuild.sh` — no APK rebuild needed for pure JS/CSS/route
changes. APK rebuilds are only required for:

- New permissions in `AndroidManifest.xml`
- New Capacitor plugins (npm packages)
- New / changed custom Kotlin under `native/android/`
- Gradle dependency changes

See `docs/CAPACITOR_BUILD.md` for the manifest, `MainActivity.java`, and
Gradle patches.
