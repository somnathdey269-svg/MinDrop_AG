# Environment Setup

All environment variables used by MindDrop.

**Rule of thumb:**
- `VITE_*` — safe for the browser bundle (public).
- Everything else — server-only. Never expose. Never rename to `VITE_*`.

Secrets are stored via Lovable Cloud (`add_secret` / `generate_secret` / `set_secret`). There is no committed `.env` with production secrets.

---

## Public Variables (browser-safe)

| Variable | Purpose | Where set |
|---|---|---|
| `VITE_SUPABASE_URL` | Supabase project URL | Auto-generated by Lovable Cloud |
| `VITE_SUPABASE_PUBLISHABLE_KEY` | Supabase anon/publishable key | Auto-generated |
| `VITE_SUPABASE_PROJECT_ID` | Project ref | Auto-generated |
| `VITE_LOVABLE_CONNECTOR_*` (if any) | Non-secret connector fields | Set by `standard_connectors--connect` |

## Private Variables (server-only)

| Variable | Purpose | How to obtain |
|---|---|---|
| `SUPABASE_URL` | Supabase URL (server) | Lovable Cloud auto |
| `SUPABASE_PUBLISHABLE_KEY` | Server publishable key | Lovable Cloud auto |
| `SUPABASE_SERVICE_ROLE_KEY` | Admin client key — RLS bypass | Managed by Lovable Cloud (not user-fetchable) |
| `SUPABASE_DB_URL` | Direct DB connection string | Managed |
| `LOVABLE_API_KEY` | AI Gateway + connector auth | Managed; rotate via `lovable_api_key--rotate_lovable_api_key` |
| `CASHFREE_APP_ID` | Cashfree merchant ID | Cashfree dashboard → App IDs |
| `CASHFREE_SECRET_KEY` | Cashfree API secret | Cashfree dashboard |
| `CASHFREE_ENV` | `SANDBOX` \| `PRODUCTION` | Set manually per environment |
| `CASHFREE_WEBHOOK_SECRET` | HMAC verification for webhook | Cashfree dashboard → Webhooks |
| `FCM_PROJECT_ID` | Firebase project ID | Firebase console → Settings |
| `FCM_SERVICE_ACCOUNT_JSON` | Full service account JSON | Firebase → Service accounts → Generate key |
| `FCM_VAPID_PUBLIC_KEY` | Web push VAPID key | Firebase → Cloud Messaging → Web push certs |
| `GOOGLE_API_KEY` | Google Maps / geocoding | Google Cloud Console |
| `GOOGLE_DRIVE_CLIENT_ID` | OAuth client ID | Google Cloud → OAuth credentials |
| `GOOGLE_DRIVE_CLIENT_SECRET` | OAuth client secret | Same |

Cross-reference with [CONNECTORS.md](./CONNECTORS.md) for what each secret unlocks.

## How to Add a Secret

- **User provides value** → `add_secret` tool (opens secure form).
- **App generates random value** (e.g. session/JWT secret) → `generate_secret` tool.
- **Value is fixed and known** (config flag) → `set_secret` tool.
- **Shared with third party** (webhook secret) → user generates locally, pastes into both the provider and `add_secret`.

Never hardcode a secret in a source file, and never invent a placeholder for a user-supplied value.

## Rotation Strategy

| Secret | Cadence | Method |
|---|---|---|
| `SUPABASE_SERVICE_ROLE_KEY` | On suspected leak | `supabase--rotate_api_keys` (platform) |
| `LOVABLE_API_KEY` | On auth errors from gateway | `lovable_api_key--rotate_lovable_api_key` |
| `CASHFREE_SECRET_KEY` | Yearly + on suspected leak | Rotate in Cashfree, then `update_secret` |
| `CASHFREE_WEBHOOK_SECRET` | Yearly | Rotate in Cashfree webhook config, then `update_secret` in same visit |
| `FCM_SERVICE_ACCOUNT_JSON` | Yearly | Firebase → Service accounts → new key; `update_secret` |
| `GOOGLE_DRIVE_CLIENT_SECRET` | Yearly | Google Cloud → OAuth → regenerate; `update_secret` |
| Signing keystore (Android) | **Never** — losing it means new package ID | Store two encrypted copies (see [MASTER_ARCHITECTURE.md](./MASTER_ARCHITECTURE.md#16-disaster-recovery)) |

## Security Guidelines

- Server secrets stay in server code (`.functions.ts` handlers, `.server.ts`, server routes).
- Read secrets **inside** the handler, not at module scope in shared files.
- Never log a secret. Never return it from a server function.
- Webhook endpoints must verify signature **before** any DB write ([SECURITY.md](./SECURITY.md#webhook-verification)).
- Publishable keys in source code are OK; anything else is not.
- On rotation, redeploy so runtime picks up new values.
- The client Supabase file (`src/integrations/supabase/client.ts`) is auto-generated — do not hand-edit to inject secrets.

## Local Development

Lovable Cloud injects env vars into the sandbox automatically. Running `bun run dev` inside the sandbox is enough — no `.env` file to manage. If you clone the repo outside Lovable, set the `VITE_*` vars minimally to boot the client; server functions will not work without the private secrets, which are only accessible through Lovable Cloud.
