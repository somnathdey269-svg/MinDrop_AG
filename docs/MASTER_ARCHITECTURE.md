Key principles:

- **One backend, many clients.** The APK never ships business logic that isn't
  also enforceable server-side.
- **Web app = feature parity minus OS integrations.** Alarms, geofencing,
  notification listener, background wake ŌĆö Android-only.
- **Marketing stays on the web.** The APK is app-facing only; no marketing
  pages, no blog, no pricing page inside the APK.
- **Server owns state, client owns UX.** Any rule that affects billing,
  quotas, or reminders is decided server-side. The client only *renders* and
  *triggers*.

## 7. Server-Driven Configuration

Anything that must change without shipping a new APK lives on the server:

- **Plan limits & quotas** ŌĆö `plans` / `plan_limits` tables
- **Feature flags** ŌĆö `feature_flags` table, read by both web and app on boot
- **Reminder rules engine** ŌĆö server function `fire-reminders` (cron)
- **Pricing** ŌĆö `products` / `prices` tables + Cashfree config
- **AI prompts & model choice** ŌĆö server-side; client just calls the fn
- **Content copy for legal / help / onboarding** ŌĆö `content_blocks` table
- **Admin console permissions** ŌĆö `user_roles` table
- **Cron schedules** ŌĆö `pg_cron` in Postgres
- **Push notification templates** ŌĆö server-side rendering before FCM send
- **Kill switches** (disable a feature globally) ŌĆö `feature_flags`

## 8. Client-Driven Configuration

- Minimum supported OS versions (`minSdk`, browser targets)

- Deep link intent filters

## 9. Features Updatable WITHOUT a New APK

## 10. Features That ALWAYS Require a New APK

Any code path that touches native OS APIs, permissions, or bundled assets:

- New Android permission (`AndroidManifest.xml`)
- New `Service`, `BroadcastReceiver`, `ContentProvider`
- Changes to AlarmManager scheduling logic
- Changes to Geofence registration
- Changes to NotificationListenerService filters (parsing rules can be server-driven; the *listener declaration* cannot)
- Cashfree native SDK version bump
- Firebase SDK version bump
- New deep link scheme
- New foreground service type
- Launcher icon / app name / package ID
- `targetSdk` / `minSdk` change
- Room schema migration
- New native module / JNI code
- Compose UI changes that aren't gated behind a feature flag+remote layout

## 13. Build Pipeline

**Web (automatic via Lovable):**
1. Push to `main` ŌåÆ Lovable build ŌåÆ published to `getmindrop.lovable.app`.
2. Push to `dev` ŌåÆ Lovable preview build.
3. Supabase migrations apply automatically on merge to `main`.

**Android (GitHub Actions):**
1. Push to `android/*` or `main` ŌåÆ workflow runs.
2. `./gradlew lint testDebug` ŌĆö unit tests + lint.
3. `./gradlew assembleRelease bundleRelease` ŌĆö signed APK + AAB.
4. Upload artifacts to GitHub Releases (draft) and Play Console Internal track.
5. Manual promotion: Internal ŌåÆ Closed ŌåÆ Open ŌåÆ Production.

Signing key lives in GitHub Actions secrets (`ANDROID_KEYSTORE_BASE64`,
`ANDROID_KEYSTORE_PASSWORD`, `ANDROID_KEY_ALIAS`, `ANDROID_KEY_PASSWORD`).
Never commit `keystore.jks` or `keystore.properties`.

## 14. Release Pipeline

**Web release:** merge `dev` ŌåÆ `main`, click Publish in Lovable. Live in
~60 seconds. No versioning ceremony required, but tag `v{n}-web` for audit.

**Android release:**
1. Bump `versionCode` and `versionName` in `android/app/build.gradle.kts`.
2. Update `CHANGELOG.md`.
3. Merge `android/release-x.y.z` ŌåÆ `main`.
4. CI produces signed AAB.
5. Upload to Play Console ŌåÆ Internal testing.
6. QA sign-off (24ŌĆō48h).
7. Promote to Closed ŌåÆ Open ŌåÆ Production (staged rollout: 10% ŌåÆ 50% ŌåÆ 100%).
8. Tag `v{n}-android`.

**Server-driven config changes** (feature flags, plan limits, prompts): change
in DB ŌåÆ live immediately. No release needed. Log the change in an audit table.

## 15. Rollback Strategy

| Layer | Rollback method | Time-to-recover |
|---|---|---|
| Web UI | Lovable version history ŌåÆ restore | < 2 min |
| Server function | Redeploy previous commit from GitHub | < 5 min |
| DB migration | Forward-only. Write a *compensating* migration. Never `DROP` in a rollback. | 5ŌĆō30 min |
| Feature flag | Toggle off in `feature_flags` table | Seconds |
| Android APK | Play Console ŌåÆ Halt rollout ŌåÆ Promote previous AAB | 15ŌĆō60 min (Play propagation) |
| Cashfree | Switch `CASHFREE_ENV` secret from PROD ŌåÆ SANDBOX | Immediate |
| Cron job | Disable in `pg_cron` (`UPDATE cron.job SET active=false`) | Seconds |

**Golden rule:** every risky release goes out behind a feature flag. Rollback =
flip the flag, not revert the code.

## 16. Disaster Recovery

**Backups:**
- Supabase Postgres: daily automated backup (Lovable Cloud managed), 7-day PITR.
- Weekly manual export to cold storage (Google Drive folder, encrypted zip).
- GitHub itself is the code backup; mirror to a second remote monthly.
- Play Console keeps every uploaded AAB indefinitely.
- Signing keystore: stored in **two** encrypted locations (1Password + offline
  USB in a safe). Losing it = losing the ability to update the app forever.

**Recovery playbook:**

| Scenario | First action | Target RTO / RPO |
|---|---|---|
| Bad web deploy | Restore previous Lovable version | 5 min / 0 |
| Bad server fn | Revert commit, redeploy | 10 min / 0 |
| DB data corruption | PITR restore to timestamp | 1 hr / <1 hr |
| Full Supabase project loss | Restore from weekly export ŌåÆ new project ŌåÆ repoint `.env` | 4 hr / 7 days worst case |
| Lost signing key | **Unrecoverable.** Publish new app under new package ID. Migrate users via deep link + server-side account portability. | Days |
| Play account suspension | Have secondary developer account pre-registered | Days |
| Cashfree outage | Show "payments temporarily unavailable" banner (server flag) | Instant |
| FCM outage | Fall back to in-app polling + AlarmManager local reminders | Instant |

## 17. Future Scalability Recommendations

**0ŌĆō10k users (today):** current stack is fine. No changes needed.

**10kŌĆō100k users:**
- Move heavy analytics to a read replica.
- Introduce a queue (Postgres `pgmq` or Cloudflare Queues) for reminder fan-out.
- Add Sentry + PostHog (already flagged in `PROJECT_AUDIT.md`).
- CDN caching for marketing routes.

**100k+ users:**
- Split marketing site into a fully static build (SSG on Cloudflare Pages).
- Move `fire-reminders` out of a single cron into a partitioned scheduler.
- Introduce Redis for hot reads (feature flags, plan limits).
- Consider dedicated Supabase compute tier or migrating hot tables to a
  managed Postgres with connection pooling (PgBouncer).
- Introduce a mobile BFF (backend-for-frontend) if native and web needs diverge.

**Cross-cutting recommendations regardless of scale:**
- Keep the "server-driven config" list in section 7 growing, not shrinking.
- Every new feature must ship behind a flag.
- Every new table must ship with RLS + GRANTs in the same migration.
- Every new native permission must be justified in `PLAY_STORE_JUSTIFICATIONS.md`.
- Keep this document updated. A stale architecture doc is worse than none.

## Duplicate-Ownership Risk Register

Areas where two owners could accidentally both edit the same thing, and how to
prevent it:

| Risk area | Who might collide | Prevention |
|---|---|---|
| **Reminder logic** in web fn *and* Kotlin AlarmManager | Backend + Android | Rule: **server decides *what* fires, Android decides *how* it wakes**. Kotlin only executes payloads from `next_reminders` table; never encodes rules. |
| **Plan limits** hardcoded in UI vs stored in DB | Lovable + Backend | Never hardcode a number in JSX/Compose. Always read from `plan_limits`. Lint rule: grep `PLAN_LIMIT` in client code, fail CI if found. |
| **Feature flags** duplicated in code as booleans | Lovable + Android | All flags flow through `feature_flags` API. No `const ENABLE_X = true` constants. |
| **Auth session handling** in web vs Kotlin | Lovable + Android | Both consume Supabase SDK. Never roll a custom JWT flow. Session refresh belongs to SDK. |
| **Push notification content** in Kotlin *and* server | Backend + Android | Server always renders the final title/body. Kotlin only displays. No client-side templating. |
| **Deep links** declared in `AndroidManifest` and web routes | Android + Lovable | Maintain `docs/DEEP_LINKS.md` as the shared spec. Both sides reference it; PRs touching either must update the doc. |
| **DB migrations** written outside the migration tool | Any | Enforce: only migrations created via Lovable's `supabase--migration` tool are accepted. CI rejects hand-written SQL in `supabase/migrations/` without the tool's header. |
| **Supabase client files** hand-edited | Any | These files carry a "DO NOT EDIT" header. CI check: fail if `src/integrations/supabase/*` differs from generator output. |
| **Cashfree keys** in both client and server | Backend | Only *publishable* key in client. Secret keys server-only. Enforced by secret naming convention (`CASHFREE_SECRET_*` never prefixed `VITE_`). |
| **Cron schedules** in both `pg_cron` and app code | Backend | Only `pg_cron`. App code never uses `setInterval` for scheduled work. |
| **Content copy** hardcoded in UI vs `content_blocks` table | Lovable + Product | Anything legal/onboarding/help goes through `content_blocks`. UI copy that never changes may stay inline. |
| **Icons / branding** in web vs Android res folders | Lovable + Android | Keep a `docs/BRAND_ASSETS/` folder as the master. Both surfaces copy from there. |
| **Analytics events** named differently on web vs Android | Both | Maintain `docs/ANALYTICS_EVENTS.md`. Event names are contract; renaming breaks funnels. |
| **User role checks** on client vs server | Everyone | Client checks are UX only. Server `has_role()` is the authority. Never trust a client role check for security. |
| **Environment config** in `.env` vs GitHub secrets vs Play Console | DevOps | One matrix in `docs/SECRETS_MATRIX.md` listing every secret and its home. Reviewed each release. |
